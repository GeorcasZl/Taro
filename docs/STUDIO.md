# Taro Studio 编辑器设计

## 设计目标

消除过去分支叙事工具中“写本本”与“排布视觉”分离的痛点。Taro Studio 的核心设计目标是：**统一的创作界面，所有编辑操作在一个协同互通的画面中完成。**

## 产品姿态

Taro Studio 呈现为一种现代、专注的专业创作软件姿态。它并非玩具式的填空工具，而是允许高度自由发挥的编辑器。创作者既可以像在普通文档编辑器里一样飞速打字，也可以像在使用设计软件一样拖拽视觉元素。

## 统一编辑界面

### 1. 剧本流（Script Flow）
这是创作者编写叙事内容（如对话、旁白、选项）的主要区域。
- **可缩放的结构树视图**：
  - **缩小**：类似地图缩放，视角拉远后，剧本流显示为全局的节点和分支结构鸟瞰图。
  - **正常**：显示当前路径上的 `Group` 列表（类似于聊天流）。
  - **放大**：聚焦单个 `Group`，查看和编辑其详细文本与设定。
- **内嵌分支**：分支选择不会跳转到一个全新的页面，而是以内嵌缩进或并列树状结构的形式直接显示在剧本流中，方便直观追踪“在这个选项之后发生了什么”。

### 2. 舞台画布（Stage Canvas）
这是视觉反馈和空间编辑区域。
- **可交互编辑**：创作者可以直接在画布上拖拽角色位置、缩放背景、设置点击热点等。
- **实时同步**：在画布上的任何操作（如将人物移到左边），都会实时转化为一条 `Document Command` 写入当前 `Group` 中。
- **所见即所得**：画布始终反映当前选中 `Group` 在其特定“路径上下文”下的最终视觉状态。

### 3. 动作时间线（Action Timeline）
负责展示和编辑当前 `Group` 内部发生的动作序列之时序关系。
- **序列概览**：以时间轴的形式，将进入该 Group 触发的一系列响应（例如：0.0s 切换背景 → 0.5s 播放音效 → 0.5s 角色入场 → 1.0s 显示对话）可视化。
- **拖拽排布**：可以直接拖拽动作块来调整它们的先后顺序或时间偏移量。
- **属性展开**：点击时间线上的任何一个动作块，即可在右侧展开其详细属性面板（例如替换音效文件、修改过渡时长）。

### 4. 工具栏（Toolbar）
提供快速的内容插入入口。
- **全局搜索（Cmd/Ctrl+K）**：Taro 的核心输入方式。按下快捷键后弹出命令菜单，可以快速添加对话、独白、分支、背景切换等任何支持的组件或动作。
- **常用快捷按钮**：常驻于界面上的图标，用于快速插入最高频的元素（如新文本、新选项）。

### 5. 素材库（Asset Library）
统一管理项目用到的所有外部资源。
- **分类管理**：角色立绘、背景图、BGM、音效、UI 贴图等分类。
- **拖拽应用**：可以将素材直接从库中拖入剧本流（生成对应事件）或拖入舞台画布（直接应用）。

### 6. 流程图视图（Flowchart View）
虽然剧本流提供了可缩放的树状结构，但在处理极端复杂、多线交叉重组的网状叙事时，纯粹的逻辑流程图仍是必要的。
- **派生视图**：流程图是不可直接编辑节点的（只能调整连线或布局），它完全基于 Document 数据派生渲染。
- **双向导航**：在流程图上双击任意节点，剧本流和画布会立刻跳转并同步到该节点的内容。

## 交互原则

1. **Document 优先**：界面上的所有操作最终都转化为针对唯一数据源的 `Document Command`。没有“仅存在于画布上”的幽灵数据。
2. **结构持续可见但从属**：复杂的网络结构是分支叙事的特点，但写作才是日常。通过滚动滚轮缩放，创作者可以随时在“写作心流”和“结构梳理”间无缝切换，结构不再干扰写作视线。
3. **就地编辑**：尽量在元素所在的位置直接进行修改，避免弹出遮挡视线的对话框模态窗口。
4. **路径上下文显式化**：在编辑任意一个深层节点时，界面上会显式提示当前是通过哪条分支路径到达这里的，确保创作者清楚当前的视觉基础是什么。
5. **诊断可导航**：当后台检测到逻辑冲突或资源丢失时，错误信息将作为跳转链接，点击即可直达案发现场。

## 输入规则

- 默认情况，回车键（`Enter`）用于在文本块中换行。
- 特定的快捷键组合（如 `Cmd+Enter`）用于快速结束当前 Group 并新建下一个 Group。
- 支持类似于 Markdown 或 Notion 的快捷斜杠指令（`/`）以快速插入不同类型的内容组件。

## 视觉语言

Taro Studio 遵循**安静、精确、编辑器风格**的视觉语言：
- **高可读性**：以内容为绝对核心，避免花哨的界面元素喧宾夺主。
- **稳定间距**：网格系统严谨，提供工程化的可信赖感。
- **适度对比**：在暗色或亮色模式下，界面框架的对比度尽量克制，让鲜艳的颜色留给创作者的美术素材。
- **轻量结构标记**：用于表示分支和缩进的线条尽量细长且低对比度。
- **强键盘亲和**：所有高频操作都必须有对应的键盘快捷键，且焦点转移自然，满足高级用户的操作效率。

## Figma 使用规范

我们使用 Figma 作为探索和讨论这些 UI 概念的原型工具。但请注意（参见 `spec/figma-workflow.md`）：Figma 中的设计稿主要用于验证**布局和交互方向**。前端最终实现时应优先考虑组件的扩展性和状态一致性，而非对 Figma 进行像素级还原。

## 布局策略

- **MVP 阶段**：采用固定的五区融合布局，降低工程复杂度，优先验证核心交互循环。
- **远期目标**：实现类似 VS Code 的工作区自定义，允许用户自由拆分、拖拽排列各面板。
