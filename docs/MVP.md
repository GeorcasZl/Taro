# Taro MVP 定义

## 目的

MVP（最小可行性产品）的目的是通过构建一个极其克制的垂直切片，来验证 Taro v2 架构中最核心的三项假设：
1. **统一编辑界面**的交互体验是否顺畅。
2. **事件驱动引擎**（trigger/response）能否正确且高效地驱动叙事。
3. **桌面应用导出**流程是否打通。

如果 MVP 成功，意味着后续所有的功能堆叠（复杂的分支、记录系统、自定义组件）都有一个极其坚固的地基。

## MVP 切面

MVP 的场景被严格限制在：**最简单的纯线性 VN 对话场景。**
通过这个场景，证明以下 4 个验证点：

1. 创作者能在**剧本流**中顺畅打字，通过输入语法糖（如 `角色名: 对话内容`），底层能自动生成正确的事件绑定。
2. 创作者能在**舞台画布**上看到这句对话的效果，并且能直接拖拽背景/角色，其修改能同步回当前 Group。
3. 创作者能在**动作时间线**中看到该 Group 内部的动作序列（显示文字、切背景），并能调整它们的先后顺序。
4. 在 Studio 内能开启**事件驱动的连贯预览**，并最终**打包导出一个可运行的桌面执行文件**（Electron/Tauri）。

## MVP 创作者工作流

在 MVP 中，创作者的完整旅程如下：

1. **新建项目**：打开 Taro Studio，建立空项目。
2. **导入素材**：将一两张背景图和立绘拖入素材库。
3. **编写对话**：在剧本流中连续敲击键盘输入对话。每次换行（特定快捷键）即生成一个新的 `Group`。
4. **舞台排布**：在某句对话处，从素材库将新背景拖入画布，画面立即更新。
5. **时序微调**：展开时间线，让“切背景”的动作比“显示对话”早 0.5 秒发生。
6. **即时预览**：点击播放按钮，从当前位置开始以玩家视角预览刚才排布的场景。
7. **一键导出**：点击导出，稍等片刻，获得一个可以直接发给朋友点击运行的 `.exe` 或 `.app` 桌面应用。

## MVP 功能边界

为了确保极速验证核心架构，MVP 必须严格遵循以下边界：

### 包含
- 基础的 Group 创建与删除。
- 基础对话语法糖及其对应的文本显示响应。
- 基础舞台操作响应（设置背景、显示角色、简单移动）。
- 纯线性的执行流（由上至下的默认顺序，无复杂跳转）。
- 事件总线的简单实现（支持 `group_enter` 和 `click` 触发器）。
- 简单的桌面应用打包导出脚本。

### 不包含
- ❌ 分支、选项、条件跳转逻辑。
- ❌ 记录系统（变量、旗标等状态管理）。
- ❌ 组件系统的注册与开放 API（先硬编码几个内置组件）。
- ❌ RPG/漫画面板等非 VN 模式。
- ❌ 面板布局自定义拖拽。
- ❌ 存档/读档机制。
- ❌ 多人协作与版本控制支持。

## MVP 实现契约

### 包结构
- `@taro/core`：只负责核心的 Document 结构和事件总线的抽象。
- `@taro/studio`：基于 React 的统一编辑器前端。
- `@taro/runtime`：执行引擎的具体实现，可挂载于 Studio 中用于预览，也可独立运行。
- `@taro/exporter`：负责将内容连同 runtime 封装为桌面应用的工具链。

### Document Schema（MVP 子集）
MVP 版本仅需实现 `taro.document.v1` 协议的最简子集，即只包含线性的 Group 数组和内嵌的 `show_text`, `set_background` 等少数事件绑定。不需要定义记录表或复杂图结构。

### 导出格式
MVP 产出的不是最终的商用级桌面应用包，而是一个能证明打包流程走通的原型：
- 包含编译后的前端 Runtime 代码。
- 包含最新的 `document.taro.json`。
- 包含被引用的少数几张图片资源。
- 捆绑在一个轻量级的 Electron 或 Tauri 壳子中。

## MVP 之后的方向

当 MVP 证明了“数据统一写入”、“视图实时协同”以及“事件引擎能够跑通”这三大支柱后，下个阶段将立刻引入**分支叙事**和**记录系统**，验证引擎在复杂图状结构中的状态派生能力。
