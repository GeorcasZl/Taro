# Taro 测试策略

## 测试理念

由于 Taro 采用事件驱动架构和多视图统一协同，我们的测试策略必须保障底层的状态机绝对可靠，以及核心用户界面的操作链路不会断裂。

## 测试分层

### 1. 引擎与核心状态机（Unit & Integration Tests）

这是测试中最重要的部分。核心库 `@taro/core` 和引擎库 `@taro/runtime` 不依赖浏览器或 DOM，可以使用纯 Node.js 测试框架（如 Vitest 或 Jest）进行测试。

**核心测试目标：**
- **命令校验与补丁**：发送非法的 `Document Command` 能否被正确拒绝？发送合法的命令是否产生了预期的 `Document` 补丁？
- **触发器与响应匹配**：给引擎发送一系列模拟事件流，断言它是否按预期激活了对应的事件绑定。
- **状态推演**：构建一个具有长路径的虚构 Document，传入不同的 `Position`，断言引擎是否能推演出完全正确且一致的舞台状态和记录状态。

*这类测试必须保持高覆盖率（>90%），并且执行速度极快。*

### 2. 统一编辑界面与组件（Component & UI Tests）

测试前端 Studio 是否正确渲染了 Document，并且用户的操作正确发出了事件。

**UI 测试目标：**
- **渲染一致性**：给定相同的 Document 状态，剧本流和时间线是否渲染出了正确的节点和动作块。
- **交互契约**：使用 React Testing Library 模拟“拖拽节点”、“修改文本”，断言 UI 是否正确发出（`dispatch`）了预期的 `Document Command`。此时不需要测试命令是否真的生效了（这由引擎层测试保障），只需测试视图发出的载荷是否合法。

### 3. 端到端工作流（E2E Tests）

使用 Playwright 等工具，在真实的浏览器或 Electron 环境下跑通整个应用循环。

**E2E 关键链路：**
- 打开应用 -> 创建项目 -> 在剧本流打一行字 -> 拖入一张背景图 -> 点击预览按钮 -> 画面正确显示并播发刚才的修改。
这类测试最脆弱也最慢，应当只保留几条覆盖上述核心“创作者工作流”的“主干路径”（Happy Paths），而不追求覆盖所有细枝末节。

## 持续集成 (CI) 要求

在提交合并请求（PR）前：
1. 必须运行所有的核心状态机测试，且必须全数通过。
2. 如果新增了一种事件或命令（如新的舞台动作），必须补充相应的状态推演和执行用例。
3. 如果修改了已有的接口契约，必须确保所有使用了该接口的内置组件一并更新并测试通过。
